2. Introduction

perfSONAR-PS is a set of independent software services that implement the perfSONAR protocols for network performance monitoring. perfSONAR-PS services are designed to be compatible with all other perfSONAR software that implements the perfSONAR Base Protocol.

The perfSONAR-BUOY Measurement Archive service exposes active measurement data, making the results available through perfSONAR web services interfaces. All performance tests are performed via the BWCTL tool (throughput measurements) and currently consist of regularly-scheduled Iperf tests to a configurable list of source and destination hosts.

Data collected through the BWCTL interaction is stored in a MySQL database. Active measurements between known hosts are particularly valuable when assuring connectivity, availability, and quality of the network. perfSONAR-BUOY allows for the easy configuration of a mesh of tests to hosts equipped with BWCTL.

Based on a configurable schedule, the tests will be conducted autonomously and stored for consumption via the perfSONAR-BUOY interface as well as presentation through included web scripts. Using the same XML protocols as the the other perfSONAR services, perfSONAR-BUOY provides uncomplicated access methods to retrieve the data in an unambiguous manner, thus eliminating the mystery associated with the backend storage.

Closely coupled with development on BWCTL, perfSONAR-BUOY will soon offer data originating from sources besides Iperf such as OWAMP (latency measurements), thrulay, and nuttcp.

3. Protocols Implemented

The perfSONAR-BUOY Measurement archive implements portions of the perfSONAR LS Protocol for service registration and management. The following messages from this protocol are supported.

  • LSRegistrationRequest

  • LSKeepaliveRequest

The only notable deviation from this protocol is the inclusion of service specific data into the LSRegistrationRequest message.

The perfSONAR-BUOY Measurement archive also implements portions of the perfSONAR MA Protocol. The first notable exception is that it does not implement the MeasurementArchiveStore message for reasons of security. Future iterations of the service will consider implementing an interface to this message type.

4. Schemas Implemented

The perfSONAR-BUOY Measurement Archive implements the [Iperf Schema Profile] in it's entirety. There are no changes to this implementation.

Future iterations of this service will consider implementation of any or all of the following schema profiles:

  • BWCTL Schema Profile

  • OWAMP Schema Profile

  • Thurlay Schema Profile

  • Nuttcp Schema Profile

  • Achievable Bandwidth Schema Profile

5. Installation

There are three primary methods used to install the perfSONAR-BUOY Measurement Archive.

  • Subversion - The perfSONAR-PS code base is available for download via our subversion server, See Subversion

  • CPAN - The preferred method of installation is through the Comprehensive Perl Archive Network, See CPAN

  • RPM - Enclosed RPM files are also available, See RPM

5.1. Prerequisites

There are several items of prerequisite software that are required before the installation of the perfSONAR-BUOY Measurement Archive. The individual instructions regarding the installation and configuration of these items is well beyond the scope of this work; users are encouraged to read the documentation of the individual software packages before proceeding further in this installation process.

  • Perl - Available on most modern Linux and Unix distributions, or see Perl

    .

  • MySQL - Open source database, see MySQL

  • AMI - The AMI framework is currently undergoing a complete re-organization. The old software may be obtained by contacting the perfSONAR-PS developers

5.2. Subversion

From a shell in the destination directory, do the following:

$ svn checkout http://anonsvn.internet2.edu/svn/perfSONAR-PS/branches/merge/

This will download the current working branch of the perfSONAR-PS repository. If you have this already it can be updated from time to time with:

$ svn update

6. Configuration

All services in the perfSONAR-PS framework run on a shared software infrastructure simply termed the perfSONAR-PS Daemon. To properly operate this, and the other services a certain amount of configuration is required. This configuration addresses both universal options that are required for the Daemon and service specific options for the perfSONAR-BUOY Measurement Archive.

There are two primary methods to configure the Daemon:

  • Script - A interactive perl based script to generate a configuration file, see perfSONAR-PS Daemon

  • Web Configuration - A web based form used to generate a configuration file and control other aspects of the software, see perfSONAR-PS WebAdmin

6.1. perfSONAR-PS Daemon

6.1.1. psConfigurDaemon

From a shell, type the following command:

$ psConfigureDaemon

The following menu will appear: 

                  
-- perfSONAR-PS Daemon Configuration --

- [press enter for the default choice] -


What file should I write the configuration to? [/etc/perfsonar/daemon.conf]:

After selecting a file, you shall see this: 

          
1) Add/Edit endpoint

2) Set global values

3) Save configuration

4) Exit

? []:

If this is your first time setting up ANY perfSONAR-PS service please configure option (2) as well as option (1) to set up this specific service. If this is an additional service to a perfSONAR-PS daemon then only worry about option (1).

A series of questions will appear, answer them and finally save the configuration using option (3) and exit using option (4).

7. Service Operation

There are different ways to run this service depending on the installation method.

7.1. Running from Subversion

The service can be started via the script in the root of the downloaded perfSONAR-PS subversion repository:

$PS_PS_HOME/./perfsonar-daemon.pl

For usage information, and options you may pass to the daemon, try this:

$PS_PS_HOME/./perfsonar-daemon.pl -h

7.2. Running from RPM or CPAN

The service can be started via the init.d facility:

$ /etc/init.d/perfsonar-daemon start

For usage information, and options you may pass to the daemon, try this:

perfsonar-daemon -h
